【红队技术/过程】

🔍 问题本质：信息≠知识，知识≠能力

  • 信息：“这是Spring Boot应用”
  • 知识：“Spring Boot有Actuator端点可能泄露信息”
  • 能力：“如何通过Actuator拿到数据库凭据并横向移动”

中间的转化需要 “攻击路径构建框架”。

🧠 四步构建法：从技术栈到攻击路径

第一步：建立你的“技术栈-攻击模式”映射库

这不是死记硬背，而是建立模式识别系统。

创建你的个人知识库（建议用Notion/Obsidian）

## Spring Boot
### 攻击面地图
1. **信息泄露**
   - `/actuator` - 端点列表
   - `/actuator/env` - 环境变量（含密码）
   - `/actuator/heapdump` - 内存转储
   - `/actuator/loggers` - 日志配置
   - `/actuator/sessions` - 会话信息

2. **RCE路径**
   - SPEL注入: `${7*7}` 在参数中
   - 反序列化: 接收Jackson/XML输入
   - JNDI注入: logback JNDI配置
   - 框架漏洞: CVE-2022-22965

3. **横向移动跳板**
   - 数据库连接字符串 (env端点)
   - 云服务凭证 (env端点)
   - 内部服务发现 (eureka客户端)

### 实战案例
- 案例1: `/actuator/heapdump` → MAT分析 → 数据库密码
- 案例2: 用户输入`${T(java.lang.Runtime).getRuntime().exec('calc')}`

关键技巧：每学一个新技术栈，就建一张这样的“攻击地图”。

第二步：基于上下文的环境分析

同样的技术栈，在不同环境中的攻击路径完全不同。

环境上下文评估框架

目标: 识别出 WordPress

# 不是直接想"WordPress漏洞"，而是分析：
环境分析:
  1. 部署位置:
    - 公有云? → 检查S3桶、元数据服务
    - 本地IDC? → 检查同网段其他服务

  2. 业务角色:
    - 官网展示? → 攻击价值低，但可能暴露员工邮箱
    - 客户门户? → 包含客户数据，高价值
    - 内部Wiki? → 可能用相同密码访问其他系统

  3. 集成组件:
    - 有WooCommerce? → 支付接口测试
    - 有BuddyPress? → 社交功能，XSS蠕虫可能
    - 有LDAP插件? → 可能拿到域凭证

# 具体攻击路径推导
if 环境 == "公有云" and 业务 == "客户门户":
    攻击路径:
      1. 检查wp-config.php备份 (常见泄露)
      2. 配置中可能有AWS密钥
      3. 用密钥访问S3桶获取客户数据
      4. 通过Lambda函数横向移动

训练方法：对每个识别出的技术栈，强制自己写三种不同环境下的攻击路径。

第三步：依赖链与信任关系分析

技术栈很少孤立存在，它们存在于依赖和信任网络中。

构建“依赖攻击树”

graph TD
    A[主站: Spring Boot] --> B[数据库: MySQL]
    A --> C[缓存: Redis]
    A --> D[消息队列: RabbitMQ]
    A --> E[文件存储: AWS S3]

    B --> B1[凭证可能在:<br/>1. application.yml<br/>2. 环境变量<br/>3. Vault]

    C --> C1[未授权访问?<br/>RCE via Lua?]

    D --> D1[管理界面暴露?<br/>默认凭证?]

    E --> E1[权限配置错误?<br/>列出所有桶]

    F[攻击路径合成] --> G[1. 通过Spring Actuator<br/>获取数据库密码]
    F --> H[2. 通过Redis未授权<br/>写SSH key到服务器]
    F --> I[3. 通过S3桶列出<br/>找到备份文件]

实际操作：

# 发现Spring Boot后的依赖链分析
1. 检查数据库: /actuator/health 看数据库状态
2. 检查缓存: 扫描 6379 端口 (Redis)
3. 检查消息队列: 扫描 5672 端口 (RabbitMQ)
4. 检查配置: /actuator/configprops

第四步：攻击路径的优先级排序

不是所有路径都值得尝试，需要基于ROI（投入产出比）排序。

使用“PIE框架”评估路径

# PIE框架: 可能性(P) × 影响(I) ÷ 成本(E) = 优先级

def evaluate_attack_path(tech_stack, path):
    """
    评估攻击路径的优先级
    """
    # 可能性 (0-10): 基于技术栈特征
    if tech_stack == "WordPress" and "plugin_vuln" in path:
        probability = 8  # WordPress插件漏洞很常见
    elif tech_stack == "Spring Boot" and "actuator" in path:
        probability = 7  # 常忘记保护

    # 影响 (0-10): 能拿到什么
    if "database_access" in path:
        impact = 10
    elif "file_read" in path:
        impact = 6

    # 成本 (1-10): 时间/隐蔽性/资源
    if "noisy_scan" in path:
        effort = 8  # 高成本，易被发现
    elif "logic_flaw" in path:
        effort = 3  # 低成本，隐蔽

    # 计算优先级
    priority = (probability * impact) / effort
    return priority

# 示例评估
paths = [
    {"name": "WordPress插件RCE", "tech": "WordPress", "type": "plugin_vuln", "gain": "shell"},
    {"name": "Spring Actuator访问", "tech": "Spring Boot", "type": "actuator", "gain": "config"},
    {"name": "业务逻辑越权", "tech": "Custom", "type": "logic_flaw", "gain": "data_access"}
]

for path in paths:
    score = evaluate_attack_path(path["tech"], path)
    print(f"{path['name']}: {score:.1f}")

🛠️ 实战训练：从识别到路径的完整推演

训练案例：识别出某系统为“Vue.js前端 + Java后端”

第一步：模式匹配

我的知识库匹配:
1. Vue.js前端 → SPA应用 → API驱动 → 重点测试API安全
2. Java后端 → 可能是Spring → 检查/actuator

第二步：环境分析

# 主动收集环境信息
1. 查看响应头: X-Powered-By: Java/11
2. 查看Cookie: JSESSIONID → 确认Java
3. 查看API结构: /api/v1/xxx → RESTful
4. 查看前端代码: 找到API调用模式

第三步：依赖链分析

假设架构:
  Vue前端 → Java API网关 → 微服务集群 → 数据库

攻击链可能:
  路径1: Vue XSS → 盗取JWT → 访问API
  路径2: API网关未授权 → 直接访问微服务
  路径3: 找到某个微服务配置错误 → 直达数据库

第四步：具体测试路径生成

# 按优先级排序的测试清单
高优先级 (高概率高影响):
  1. 测试API越权: 修改user_id参数
  2. 测试JWT安全: 修改算法、过期时间
  3. 寻找/actuator端点

中优先级:
  4. Vue XSS测试: {{7*7}} 在输入点
  5. API参数注入: SQLi、命令注入
  6. 文件上传点测试

低优先级 (需要更多侦察):
  7. 寻找微服务单独暴露
  8. 寻找管理后台
  9. 测试WebSocket连接

📚 构建你的“攻击路径思维”训练计划

每周训练任务

第1-2周：基础模式积累

  • 选择3个常见技术栈（如：Spring Boot、WordPress、React）
  • 为每个建立完整的“攻击面地图”
  • 收集至少5个真实案例对每个技术栈

第3-4周：环境上下文训练

  • 对同一技术栈（如WordPress），设想3种不同环境：
  • 企业官网
  • 电子商务站
  • 内部Wiki系统
  • 为每个环境写出不同的攻击路径

第5-6周：依赖链分析训练

  • 选择一个复杂系统（如：微服务电商平台）
  • 画出完整的组件依赖图
  • 找出3条从边缘到核心的攻击链

第7-8周：优先级评估训练

  • 用PIE框架评估10个不同攻击路径
  • 在CTF或演练环境中实际验证评估结果

🚨 常见陷阱与解决方案

陷阱1：信息过载，不知从何开始

症状：识别出10个技术组件，陷入分析瘫痪 解决：用 “核心-边缘”策略： - 先找核心业务功能的技术栈 - 再找防护最弱的技术栈 - 最后考虑边缘组件

陷阱2：路径单一，不懂变通

症状：只想到最明显的攻击方式 解决：强制三路径原则： - 为每个技术栈必须想3条不同攻击路径 - 1条直接漏洞利用 - 1条配置错误利用 - 1条业务逻辑绕过

陷阱3：忽视时间维度

症状：不考虑攻击的时机和顺序 解决：攻击阶段规划： - 第一阶段：隐蔽侦察（不触发告警） - 第二阶段：初步入侵（获取立足点） - 第三阶段：横向移动（扩大控制） - 第四阶段：目标达成（获取数据/权限）

✅ 检验标准：如何知道你已经掌握了

当你看到技术栈时，能在10分钟内：

  1. 说出至少3个该技术的常见错误配置
  2. 设计至少2条不同方向的攻击路径
  3. 评估哪条路径最隐蔽且最有效
  4. 明确第一步测试动作是什么

示例检验：

看到 "Nginx + PHP-FPM + MySQL" 后:
1. 常见错误: Nginx配置错误导致源码泄露、PHP-FPM未授权访问、MySQL弱密码
2. 路径1: 找Nginx配置错误 → 获取源码 → 找数据库密码
   路径2: 测试PHP-FPM RCE (CVE-2019-11043)
3. 最隐蔽: 路径1，因为只是读取文件
4. 第一步: 检查是否存在 /phpinfo.php、/test.php、备份文件

🎯 最终心法

构建攻击路径的能力不是记住所有漏洞，而是：

理解技术如何被滥用，信任如何被破坏，系统如何被串联。

这就像学下棋：开始你记“开局定式”（技术栈攻击模式），然后学“中局战术”（环境上下文分析），最后掌握“全局战略”（依赖链与路径规划）。
